# 当地コース別成績モジュール 設計ドキュメント

**最終更新**：実際のKファイル（`K260707.TXT`）での検証、およびスキーマバージョン管理・自動マイグレーション機能の追加を反映済み。

## 0. 前提・スコープ

**本モジュールは既存のBot（`notify_arashi.py`・`x_ranking.py`・`x_asahi_scoring.py`等）には一切組み込まれていません。** 完全に独立したファイル群として実装されており、既存コードは1行も変更していません。既存Botから利用する場合は、将来の別フェーズで明示的な統合作業を行います。

`k_race_history.csv`が**唯一の正（Single Source of Truth）**です。`local_course_stats.csv`はそこから毎回まるごと再生成される派生データであり、`local_course_stats.csv`自体を手で編集しても次回実行時に無条件で上書きされます。

## 2. `k_race_history.csv` の仕様（唯一の正）

### 2-1. 各カラムの意味・型・更新ルール

Kファイルから抽出した「選手1人 × 1レース」を1行とする、**追記専用（append-only）**の生データです。一度書き込んだ行の値を後から書き換えることはありません（同じ`(date, venue_code, race_no, racer_no)`の組み合わせが再度渡された場合は、新しい値で上書きするのではなく「重複としてスキップ」します。詳細は2-2節）。

| # | 列名 | 型 | 意味 | 値の由来（Kファイル上の位置） | 更新ルール |
|---|---|---|---|---|---|
| 1 | `date` | int | 開催日（YYYYMMDD） | 場ブロック内の「第N日 YYYY/ M/ D」行から抽出 | レース結果確定時に1回だけ書き込まれる。以後変更されない |
| 2 | `venue_code` | str（2桁） | 場コード（"01"〜"24"） | `"XXKBGN"`マーカーの2桁 | 同上 |
| 3 | `venue_name` | str | 場名（例: "大村"） | `venue_code`から`VENUE_CODE_MAP`で変換した表示用の値（Kファイル本文には存在しない導出値） | 同上 |
| 4 | `race_no` | int | レース番号（1〜12） | レース見出し行（例: `"  1R       予選"`）の数字部分 | 同上 |
| 5 | `racer_no` | str | 選手登録番号（4桁） | 選手別詳細行の固定長カラム`[8:12]` | 同上 |
| 6 | `racer_name` | str | 選手名（参考情報） | 選手別詳細行の固定長カラム`[13:21]`（全角スペース除去済み） | 同上。表記ゆれ（結婚等による改姓）があっても過去の行は書き換えない |
| 7 | `boat_no` | int | 艇番（1〜6、出走表上の枠番） | 選手別詳細行の固定長カラム`[6]` | 同上 |
| 8 | `course` | int（1〜6） | **進入コース**（実際にどのコースから進入したか） | 選手別詳細行の固定長カラム`[38]` | 同上。**当地コース別成績の集計キーの中核** |
| 9 | `order` | int または str | 着順。1〜6の数値、または欠場等の異常記号（例: `"K0"`） | 選手別詳細行の固定長カラム`[0:4]` | 同上。異常記号のまま保存し、集計時に数値着順のみ扱う（4節参照） |
| 10 | `motor_no` | int または空 | モーター番号 | 選手別詳細行の固定長カラム`[22:24]` | 同上。取得できない場合は空 |
| 11 | `boat_equip_no` | int または空 | ボート番号 | 選手別詳細行の固定長カラム`[27:29]` | 同上 |
| 12 | `exhibition_time` | float または空 | 展示タイム | 選手別詳細行の固定長カラム`[31:35]` | 同上。欠場時は数値化できず空になる |
| 13 | `start_timing` | float または空 | 実際のレースでのスタートタイミング（ST） | 選手別詳細行の固定長カラム`[43:47]` | 同上 |
| 14 | `race_time` | str または空 | レースタイム | 選手別詳細行の固定長カラム`[52:]` | 同上。欠場・失格時は`" .  . "`のような空値パターンになり`None`として保存 |
| 15 | `source_file` | str | 元になったKファイル名（トレーサビリティ用） | パース実行時のファイル名 | 同上。同じレースを別ファイルから重複取得した場合の追跡に使う |

**カラム位置の出典**：上記の固定長カラム位置は、実際のKファイル（`K260707.TXT`、2026/7/7開催・12場・169KB）で1文字ずつ検証済みです。ミッドナイト開催のファイルで確認したものであり、通常開催のKファイルで同一レイアウトかは未確認です。将来別のレイアウトのKファイルで抽出件数が異常に少ない場合は、まず`x_kfile_race_parser.py`の`_is_racer_line()`・固定長カラム位置がずれていないか確認してください。

### 2-2. 重複防止ルール（二重登録防止）

重複判定キーは **`(date, venue_code, race_no, racer_no)`** の組み合わせです。このキーが既に`k_race_history.csv`に存在する行は、新しいレコードが渡されても**追加されません**（上書きもされません）。これにより「同じKファイルを誤って2回処理しても安全」「同じ日付のデータを重複登録しない」という要件を満たしています。

### 2-3. 既知の欠損パターン

欠場・フライング等（着順欄が`"K0"`のような記号）の場合、Kファイル上で本来「展示タイム」の位置に`"K ."`のような記号が入り、それ以降の全カラム（進入コース・ST・レースタイム）の文字位置がずれます。この場合、**進入コースが数字として読み取れないため、そのレコード自体が抽出対象から除外されます**（`k_race_history.csv`に一切記録されません）。「当地コース別成績」には進入コース情報が必須のため、これは意図した安全動作です。実データ（`K260707.TXT`、864件中）でこのパターンが1件発生し、正しく除外されることを確認済みです。

## 3. スキーマバージョン管理（将来カラムを追加する手順）

`k_race_history.csv`の列構成は`k_race_history_schema.py`が**単一の真実の源**として管理します。列を追加・変更したくなったときは、以下の手順だけで安全に移行できます。

1. `k_race_history_schema.py`の`CURRENT_SCHEMA_VERSION`をインクリメントする（例: 1 → 2）
2. `SCHEMA_COLUMNS`辞書に、新バージョンの**完全な列リスト**を追加する
3. `NEW_COLUMN_DEFAULTS`に、新しく増えた列の**初期値**（旧データを補完する値）を追加する
4. `MIGRATIONS`辞書に`(旧バージョン, 新バージョン): 変換関数`を1行登録する

これ以降、`x_local_course_stats.py`を実行するたびに以下が自動で行われます。

- 実ファイルのヘッダーと`CURRENT_SCHEMA_VERSION`を比較し、古ければ自動でマイグレーションを実行
- マイグレーション実行前に`k_race_history_backup_YYYYMMDD_HHMMSS.csv`を自動作成（失敗時はこのバックアップから復元）
- 新しい列は`NEW_COLUMN_DEFAULTS`の値で補完し、既存データは一切失われない

**手動でCSVを直接編集する運用は禁止**です。必ずこのマイグレーション経由で変更してください。

**検証済みの実績**：`weather`列を追加するシナリオを実際にシミュレートし、既存864件のデータを一切失わずに新しいスキーマへ移行できることを確認しています（バックアップ生成・新列の初期値補完・新規追記データが新スキーマで書き込まれることも含め検証済み）。

## 4. `local_course_stats.csv` の仕様（履歴から再生成される派生データ）

### 4-1. 各カラムの意味・型

| 列名 | 型 | 意味 | 算出方法 |
|---|---|---|---|
| `racer_no` | str | 選手登録番号 | `k_race_history.csv`の`racer_no`をそのまま |
| `venue_code` | str | 場コード | 同上の`venue_code` |
| `venue_name` | str | 場名 | 同上の`venue_name` |
| `course` | int（1〜6） | コース | 同上の`course` |
| `starts` | int | 出走数（そのレーサー・その場・そのコースへの進入回数） | `(racer_no, venue_code, course)`が一致する行数（着順が異常記号の行も出走数には含む） |
| `first`〜`sixth` | int | 1着〜6着の回数 | `order`列が数値かつ該当着順の行数（異常記号の行は着順集計には含まれない） |
| `first_rate` | float（%） | 当地コース1着率 | `first / starts * 100` |
| `top2_rate` | float（%） | 当地コース2連対率 | `(first + second) / starts * 100` |
| `top3_rate` | float（%） | 当地コース3連対率 | `(first + second + third) / starts * 100` |
| `last_updated` | int（YYYYMMDD） | この行を最後に更新した日付 | 集計実行時に`--today`で渡した日付（省略時は`0`） |

### 4-2. 更新ルール

`local_course_stats.csv`は`k_race_history.csv`全体から**毎回まるごと再計算**されます（差分更新ではなく完全再生成）。実行のたびに一時ファイルへ書き出してから`os.replace`でアトミックに置き換えるため、集計処理の途中で失敗しても既存の`local_course_stats.csv`が破損することはありません。

**なぜ差分更新ではなく毎回全体再集計するのか**：この集計は「選手×場×コース」という組み合わせ単位であり、1日分の新しいレコードがどの組み合わせの集計に影響するかを追跡するロジックは複雑になりがちです。他方、集計対象データ（`k_race_history.csv`）は当面数十MB程度に収まる規模のため、「シンプルさ」を優先し、追記のたびに全体を読み直して再集計する設計にしています。将来データ量が数百MB〜GB規模に育った場合は、差分更新方式への切り替えを検討してください。

## 5. 処理フロー

```
[Kファイル(テキスト形式, .TXT)]
        │
        ▼
x_kfile_race_parser.parse_k_race_file()
  → 1ファイルにつき「選手×レース」のレコードリストを返す
        │
        ▼
x_local_course_stats.append_to_history()
  → ①k_race_history_schema.ensure_schema() でスキーマチェック
    （古ければ自動マイグレーション、判定不能なら中止）
  → ②k_race_history.csv に重複除去して追記
        │
        ▼
x_local_course_stats.rebuild_local_course_stats()
  → ①k_race_history_schema.ensure_schema() でスキーマチェック
  → ②k_race_history.csv 全体を集計し、
    local_course_stats.csv を洗い替え生成
```

## 6. 初回（2023年〜現在）の一括構築方法

```bash
python x_local_course_stats.py --init --k-dir /path/to/k_files_folder
```

- `--k-dir`で指定したフォルダ内の全Kファイル（`K??????.TXT`形式）を日付順に読み込み、`k_race_history.csv`をゼロから構築します。
- 処理は1ファイルずつ独立して行い、`k_race_history_progress.json`に「処理済みファイル名の一覧」を記録します。途中で中断しても、再実行時に未処理のファイルだけを続きから処理します（同じフォルダで再実行しても安全なことを検証済みです）。
- 全ファイル処理完了後、最後に一度だけ`rebuild_local_course_stats()`を実行します。

## 7. 毎日の更新方法

```bash
python x_local_course_stats.py --daily --k-file /path/to/K250708.TXT
```

- 前日分のKファイル1つを渡します。
- 履歴への追記後、`local_course_stats.csv`を再生成します。
- 誤って同じファイルを2回実行しても、重複除去キーにより安全です（データが二重に増えません）。

## 8. 集計のみ再実行したい場合

```bash
python x_local_course_stats.py --rebuild-only
```

`k_race_history.csv`には触れず、`local_course_stats.csv`だけを最新の内容で再生成します。

## 9. 「当地」の判定方法（重要な設計判断）

「当地」とは、選手の登録上の所属地（支部）ではなく、**「そのレースが開催された場」そのもの**を指すと解釈しています（`venue_code`列がそれに対応）。つまり「当地コース別成績」は正確には「この選手がこの場のこのコースに来たとき、過去どうだったか」という『場所×コース』ごとの成績であり、選手の出身地・所属支部とは無関係です。

## 11. 将来AIに使えそうな特徴量の提案（Kファイルの他項目より）

`k_race_history.csv`には、当地コース別成績以外にも将来使えそうな項目を既に含めています。

| 項目 | 今のうちに保存している理由 |
|---|---|
| モーター番号・展示タイム | 既存の`motor_history.csv`と同じ粒度のデータを独立して蓄積。将来「当地×モーター」のクロス集計に発展可能 |
| 実際のスタートタイミング（ST） | fanファイルの「平均ST」は期別集計値だが、こちらは1レースごとの生ST。当地コースでの実際のST分布（安定度・ばらつき）という粒度の特徴量が作れる |
| 着順記号（欠場等） | 進入コース別のフライング率・出遅れ率を、当地×コース単位で算出できる可能性 |
| レースタイム | 会場コンディションによる走破タイム傾向の分析可能性（展示・気象を予想に使わない既存方針との整合は要検討） |
