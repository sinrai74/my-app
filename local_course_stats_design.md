# 当地コース別成績モジュール 設計ドキュメント

## 0. 前提・スコープの確認

**本モジュールは既存のBot（`notify_arashi.py`・`x_ranking.py`等）には一切組み込みません。** 完全に独立した2つの新規Pythonファイルとして実装し、既存コードは1行も変更しません。既存のBotから利用する場合は、将来の別フェーズで明示的な統合作業を別途行います。

### 重要な確認事項（設計前に必ず共有すべき事実）

添付いただいたスクリプトの`parse_k_file`関数、および本プロジェクト既存の`parsers.py`の`parse_k_file`関数は、どちらも**Kファイル内の「3連単の結果行」（払戻金計算用の1行、例: `1R  1-2-3  12340`）のみ**を正規表現で抽出しており、**選手別の詳細行（着順・艇番・登番・進入コース等）は一切パースしていません**。

今回ご要望の「選手登録番号・艇番・進入コース・着順」を取得するには、**Kファイル内の選手別詳細行を新たに解析するロジック**が必要です。

競艇公式サイトが配布する「テキスト形式のKファイル」には、一般的に以下のような選手別詳細行が含まれています（一般に公開されている知見に基づく想定レイアウトです）。

```
21KBGN
                            ボートレース○○成績  20240115  1日目
                                                            1R  予選
------------------------------------------------------------------------------
 着  艇  登番  選手名           Ｍｏｔ  Ｂｏａｔ  展示   進入  ＳＴ    レースタイム
  1   3  4444  ヤマダ タロウ      22     45     6.78    3   .13     1.51.2
  2   1  3333  タナカ イチロウ    15     12     6.85    1   .15     1.52.0
  ...
```

**この行フォーマット（列の区切り位置・空白幅）は実際のファイルを見て検証していないため、設計段階では「推測に基づく初期実装」とします。** 実装後、お手元のKファイルを1つサンプルとして共有いただければ、パースロジックの列位置・正規表現を実データに合わせて調整します（これは今回の設計の中で最も重要なリスク要因です）。

以降の設計は、この前提（テキスト形式Kファイル、選手別詳細行のパースが必要）に基づきます。

## 2. CSV設計

### 2-1. `k_race_history.csv`（生データ蓄積、追記型・永続保存）

Kファイルから抽出した「選手1人×1レース」を1行として、日々追記していく生データです。将来の別集計（当地コース別成績以外の分析）にも再利用できるよう、着順記号・モーター・展示タイム等も併せて保存します（詳細は5節）。

| 列名 | 型 | 説明 |
|---|---|---|
| `date` | int | 開催日（YYYYMMDD） |
| `venue_code` | str | 場コード（2桁、"01"〜"24"） |
| `venue_name` | str | 場名（例: "桐生"） |
| `race_no` | int | レース番号（1〜12） |
| `racer_no` | str | 選手登録番号 |
| `racer_name` | str | 選手名（パースできた場合のみ、参考情報） |
| `boat_no` | int | 艇番（1〜6、出走表上の枠番） |
| `course` | int | 進入コース（1〜6。実際に何コースから進入したか） |
| `order` | int or str | 着順（1〜6の数値、または"F"・"L"・"K"・"S"等の異常着順記号） |
| `motor_no` | int | モーター番号（取得できれば） |
| `boat_equip_no` | int | ボート番号（取得できれば） |
| `exhibition_time` | float | 展示タイム（取得できれば） |
| `start_timing` | float | スタートタイミング（実際のレースでのST） |
| `race_time` | str | レースタイム（取得できれば） |
| `source_file` | str | 元になったKファイル名（トレーサビリティ用） |

**追記型・重複除去キー**：`(date, venue_code, race_no, racer_no)`の組み合わせで重複を判定し、同じレコードは二重に追加しません（これまでの`hit_record.csv`・`motor_history.csv`と同じ設計パターンです）。

### 2-2. `local_course_stats.csv`（集計結果、`k_race_history.csv`から再生成）

ご要望通りの列構成です。

| 列名 | 型 | 説明 |
|---|---|---|
| `racer_no` | str | 選手登録番号 |
| `venue_code` | str | 場コード |
| `venue_name` | str | 場名 |
| `course` | int | コース（1〜6） |
| `starts` | int | 出走数（そのレーサー・その場・そのコースでの進入回数） |
| `first` | int | 1着数 |
| `second` | int | 2着数 |
| `third` | int | 3着数 |
| `fourth` | int | 4着数 |
| `fifth` | int | 5着数 |
| `sixth` | int | 6着数 |
| `first_rate` | float | 当地コース1着率（%） |
| `top2_rate` | float | 当地コース2連対率（%、1着+2着） |
| `top3_rate` | float | 当地コース3連対率（%、1着+2着+3着） |
| `last_updated` | int | この行を最後に更新した日付（YYYYMMDD） |

`local_course_stats.csv`は`k_race_history.csv`全体から**毎回まるごと再計算**します（差分更新ではなく完全再生成）。これは実装をシンプルに保つための意図的な設計判断です。理由は4節で説明します。

## 3. 処理フロー

```
[Kファイル(テキスト形式, .TXT)]
        │
        ▼
x_kfile_race_parser.parse_k_race_file()
  → 1ファイルにつき「選手×レース」のレコードリストを返す
        │
        ▼
x_local_course_stats.append_to_history()
  → k_race_history.csv に重複除去して追記
        │
        ▼
x_local_course_stats.rebuild_local_course_stats()
  → k_race_history.csv 全体を集計し、
    local_course_stats.csv を洗い替え生成
```

---

## 4. 初回（2023年〜現在）の一括構築方法

```bash
python x_local_course_stats.py --init --k-dir /path/to/k_files_folder
```

- `--k-dir`で指定したフォルダ内の全Kファイル（`K??????.TXT`形式のファイル名を想定）を日付順に読み込み、`k_race_history.csv`をゼロから構築します。
- 2023年1月〜現在まで、ファイル数は概算で `24場 × 平均10レース/日 × 約365日 × 3年分 ≈ 26万レース分` になり得ますが、1レコードは軽量（数十バイト程度）なので、CSV全体でも数十MB程度に収まる想定です。
- 処理時間短縮のため、パース処理は1ファイルずつ独立して行い、途中で失敗しても「どのファイルまで処理済みか」を`k_race_history_progress.json`（進捗ファイル、このモジュール専用）に記録し、再実行時に続きから再開できるようにします。
- 全ファイル処理完了後、最後に一度だけ`rebuild_local_course_stats()`を実行し、`local_course_stats.csv`を生成します（毎ファイルごとに集計し直すと非効率なため、初回構築時は「蓄積が全部終わってから1回だけ集計」という設計にします）。

## 5. 毎日の更新方法

```bash
python x_local_course_stats.py --daily --k-file /path/to/K250708.TXT
```

- 前日分のKファイル1つだけを渡します（既存のKファイル取得スクリプトが日次で1ファイルずつ生成する想定に合わせています）。
- `append_to_history()`で`k_race_history.csv`に前日分を追記します（重複除去つきなので、誤って同じファイルを2回実行しても安全です）。
- 追記後、`rebuild_local_course_stats()`を実行し、`local_course_stats.csv`を再生成します。

**なぜ差分更新ではなく毎回まるごと再集計するのか**：`local_course_stats.csv`は「選手×場×コース」という組み合わせ単位の集計であり、1日分の新しいレコードが、どの組み合わせの集計に影響するかを追跡するロジックは複雑になりがちです。他方、集計対象データ（`k_race_history.csv`）は数十MB程度に収まる規模のため、**「シンプルさ」を優先し、追記のたびに全体を読み直して再集計する**設計にしています。将来データ量が数百MB〜GB規模に育った場合は、差分更新方式への切り替えを検討してください（このドキュメントに設計判断の理由を明記しておくことで、将来の変更判断がしやすくなります）。

## 6. 「当地」の判定方法（重要な設計判断）

ご要望の「当地」とは、**選手の登録上の所属地（支部）ではなく、「そのレースが開催された場」そのもの**を指すと解釈しています（`local_course_stats.csv`の`venue_code`列がまさにそれです）。つまり「当地コース別成績」は、正確には**「この選手がこの場のこのコースに来たとき、過去どうだったか」という『場所×コース』ごとの成績**であり、選手の出身地・所属支部とは無関係です。この解釈で問題ないか、実装前に確認させてください。

## 8. 将来AIに使えそうな特徴量の提案（Kファイルの他項目より）

Kファイルの選手別詳細行には、今回の「当地コース別成績」以外にも、以下のような特徴量に発展させられる情報が含まれています。今回は**保存だけ**しておき、活用は将来判断することを推奨します（`k_race_history.csv`の列設計に既に織り込み済みです）。

| 項目 | 今のうちに保存すべき理由 |
|---|---|
| **モーター番号・展示タイム** | 既存の`motor_history.csv`と同じ粒度のデータを、Kファイル側からも独立して蓄積できます。将来「当地×モーター」のようなクロス集計特徴量に発展させられます。 |
| **実際のスタートタイミング（ST）** | fanファイルの「平均ST」は期別の集計値ですが、Kファイルは**1レースごとの生ST**です。「当地コースでの実際のST分布（安定度・ばらつき）」という、fanファイルにはない粒度の特徴量が作れます。 |
| **着順記号（F/L/K/S等）** | 進入コース別のフライング率・出遅れ率を、当地×コース単位で算出できます。Ver4で追加した「F率/L率」（fanファイル由来、期別集計）を、**当地限定でより直近の生データから補強**できます。 |
| **レースタイム** | 会場のコンディション（潮位・水面状態）による走破タイムの傾向を、日付と紐づけて分析できる可能性があります（ただし展示・気象情報を予想に使わないという既存Botの方針と整合させる必要があるため、慎重な扱いが必要です）。 |

これらは全て`k_race_history.csv`の列に既に含めているため、**今回の実装だけで将来の拡張余地を確保**できます。

## 9. 動作確認・検証方針

1. まずお手元のKファイルを1つ（できれば直近の日付のもの）共有いただき、実際の行フォーマットに合わせて`x_kfile_race_parser.py`のパースロジックを検証・調整します。
2. その後、小規模な期間（例: 直近1週間分）で試験的に`k_race_history.csv`・`local_course_stats.csv`を生成し、既存の`fanファイル`の全国コース別1着率と数値の傾向が大きく矛盾しないか（極端な異常値がないか）を確認します。
3. 問題なければ、2023年1月〜現在までの初回一括構築を実行します。
