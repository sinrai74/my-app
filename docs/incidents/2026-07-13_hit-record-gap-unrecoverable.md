# インシデント記録: hit_record.csv 欠落期間の調査結果（復旧不能確定）

記録日: 2026-07-13
関連: Phase0.5設計書 v1.1 ⑱（Phase1開始条件）、⑥（保存層責務）

## 対象

hit_record.csv における 2026-07-05・2026-07-06・2026-07-07 の3日分のレコード欠落。

## 調査内容と結果

### 1. 現行 hit_record.csv 本体
`findstr` で各日付の存在を確認。

| 日付 | 結果 |
|---|---|
| 20260704 | 存在（多数のレコードあり） |
| 20260705 | **0件** |
| 20260706 | **0件** |
| 20260707 | **0件** |

### 2. ローカルバックアップ3種の中身確認

ファイル名の日付（作成日時）と実際に含まれるレコードの日付範囲は一致しなかった。

| ファイル名 | 作成日時（ファイル名より） | 実際の中身の日付範囲 | 件数 |
|---|---|---|---|
| hit_record_backup_20260704_215943.csv | 2026-07-04 21:59 | 未特定（別調査） | 139 |
| hit_record_backup_20260705_151657.csv | 2026-07-05 15:16 | 未特定（別調査） | 229 |
| hit_record_backup_20260705_202325.csv | 2026-07-05 20:23 | **20260701〜20260704のみ** | 229 |

3ファイルとも `findstr /c:"20260705"` および `/c:"20260706"` が0件。
つまり作成時刻が7/5であっても、その時点のhit_record.csvはまだ7/4分までしか記録が進んでいなかったことを示す。

### 3. GitHub Releases（`data-store` タグ）

Assetsに含まれるファイルを確認。

- buyscore_log.jsonl (+.bak)
- daily_stats.json (+.bak)
- motor_history.csv (+.bak)

**`hit_record.csv` はそもそも退避対象に含まれていない。** Phase0設計書で「長期データはGitHub Releases経由」としていた認識と、実際の運用実装（`x_release_storage.py`の退避対象リスト）に乖離があったことが判明した。

### 4. git履歴

`hit_record.csv` は `.gitignore` 対象のためcommit履歴からの復元も不可。

## 結論

**2026-07-05〜2026-07-07の3日分のhit_record.csvは復旧不能と確定する。**

調査したすべての保存先（本体・ローカルバックアップ・GitHub Releases・git履歴）のいずれにも当該期間のデータは存在しない。

## 副次的な発見（根本原因）

`x_release_storage.py` の退避対象リストに `hit_record.csv` が含まれていなかったことが、
今回の欠損を防げなかった直接の原因と推定される。ローカルにのみ存在し、
Actions実行環境がステートレスであるため、コンテナ終了とともにその日の記録が失われた
可能性が高い（設計書⑥「GitHub Actionsコンテナはステートレス」の原則が実運用で
守られていなかったケース）。

## 対応方針

1. 本件は「復旧不能な欠落期間」として確定し、これ以上の復旧作業は行わない
2. ⑭ゴールデンデータの選定において、2026-07-05〜07を対象期間から除外する
3. 新システムのstorage層（⑥）実装時、hit_record.csvをGitHub Releasesの退避対象に
   必ず含めることを実装要件とする（Phase1以降のstorage層設計で担保）
4. 付随して発見された `retroactive_fix:amount_zero_ghost_purchase` タグ（2026-07-04分の
   全レコードに付与）については、原因・経緯とも運用者に心当たりがないことを確認した。
   本件の復旧可否の結論には影響しないため、「原因不明の既知の異常値」として本記録に
   留め置く。今後同種の事象が再発した場合の参照情報とする
